# Palantir K8s GUI - Phase 3 Postmortem (Terminal & Session)

## 1. 중요한 점 (Key Takeaways)
* **비동기 런타임의 안전성**: Tauri의 이벤트 핸들러는 Tokio 런타임 위에서 동작하므로, 스레드를 차단하는(Blocking) 호출이 하나라도 포함되면 전체 앱이 패닉에 빠질 수 있음을 확인했습니다.
* **자원 생명주기(Lifecycle) 관리**: GUI 앱에서 실시간 스트림을 다룰 때는 단순히 "여는 것"보다, 창을 닫거나 컴포넌트가 사라질 때 "어떻게 안전하게 닫는가"가 시스템 안정성에 더 큰 영향을 미칩니다.
* **이벤트 채널의 격리**: 동일한 파드에 대해 여러 터미널을 열 수 있으므로, `PodID`가 아닌 고유한 `SessionID`를 기반으로 통신 채널을 분리해야 데이터 혼선을 막을 수 있습니다.

## 2. 잘된 점 (What Went Well)
* **실환경 통합 테스트 수행**: `kind` 클러스터를 직접 구축하여 백엔드 로직을 검증함으로써, 이론상으로는 맞지만 실제 API 서버가 거부하는 설정(`tty`와 `stderr` 동시 활성화 불가)을 사전에 찾아내어 수정했습니다.
* **독립적인 코드 리뷰 활용**: `generalist` 에이전트를 통한 교차 검증으로 세션 누수(Listener Leak)와 좀비 프로세스 문제를 배포 전에 미리 식별해 냈습니다.
* **GUI 논리 검증 도입**: `Vitest`를 사용하여 GUI가 없는 환경에서도 프론트엔드의 상태 변화와 API 호출 로직이 완벽하게 동작함을 입증했습니다.

## 3. 잘못된 점 / 실수했던 부분들 (Mistakes & Pitfalls)
* **비동기 핸들러 내 Blocking 호출**: `mpsc::blocking_send`를 사용해 런타임 패닉을 유발했습니다. 비동기 환경에서는 항상 `try_send`나 `send().await`를 사용해야 한다는 기본 원칙을 간과했습니다.
* **MutexGuard와 Await의 충돌**: 세션 종료 로직에서 락(`Lock`)을 쥐고 있는 상태로 `await`을 호출하여 `Send` 트레이트 에러(컴파일 에러)를 겪었습니다.
* **사용자 피드백 지연**: 터미널이 "Connecting" 상태에서 멈춰있을 때, 내부적으로 어떤 일이 일어나는지(통신 중인지, 에러인지) 시각적 피드백을 충분히 제공하지 못해 혼란을 드렸습니다.

## 4. 개선할 점 / 피드백 (Areas for Improvement)
* **에러 메시지의 구체화**: 단순한 "Connection Error" 대신, K8s API의 상세 응답이나 네트워크 상태를 터미널 화면에 직접 출력해 주어야 사용자가 자가 진단을 할 수 있습니다.
* **자동 재연결 로직**: 일시적인 네트워크 순단 시 세션을 끊지 않고 자동으로 다시 `attach`를 시도하는 로직이 추가된다면 더 강력한 툴이 될 것입니다.
* **UX 디테일**: 접속 성공 시 자동으로 엔터 키를 입력하여 프롬프트를 띄워주는 것과 같은 작은 배려가 사용자 경험을 크게 개선함을 배웠습니다.

## 5. 실수하기 쉬운 부분에 대한 보완 제안 (Preventative Measures)
1. **K8s Exec API의 특이점**
   * **문제**: `tty: true` 설정 시 `stderr` 스트림은 `stdout`으로 통합되며, `stderr()` 메서드는 `None`을 반환합니다. 이를 강제로 언래핑(`unwrap`)하면 패닉이 납니다.
   * **보완**: 스트림 읽기 루프를 짤 때 `stderr`의 존재 여부를 조건문으로 체크하거나, TTY 모드에 따라 전용 핸들러를 사용해야 합니다.
2. **Tauri 이벤트 페이로드의 직렬화**
   * **문제**: Rust에서 `Vec<u8>`을 보내면 JS에서는 숫자 배열(`[10, 20, ...]`)로 들어옵니다. 이를 단순히 문자열로 처리하려다 데이터가 깨질 수 있습니다.
   * **보완**: 프론트엔드에서 `Uint8Array`를 사용해 바이트를 복원하고, `xterm.js`의 `write()` 메서드에 직접 전달하여 스트리밍 디코딩을 활용하는 것이 가장 안전합니다.
3. **좀비 프로세스 방지**
   * **문제**: 터미널 창만 닫고 백엔드 세션을 종료하지 않으면 K8s API 서버와의 연결이 계속 유지되어 리소스가 낭비됩니다.
   * **보완**: 프론트엔드의 `useEffect` cleanup 함수에서 반드시 백엔드 종료 커맨드(`stop_session`)를 호출하도록 강제하고, 세션 매니저를 통해 주기적으로 고립된 세션을 청소하는 로직이 필요합니다.
