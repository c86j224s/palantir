# Palantir K8s GUI - Phase 1 & 2 Postmortem

## 1. 중요한 점 (Key Takeaways)
* **초기 아키텍처의 중요성**: `Tauri`를 브릿지로 사용하여 순수 K8s 비즈니스 로직(`core`)과 프론트엔드(`frontend`)를 완전히 분리한 설계는 향후 유지보수와 타 프레임워크(C# 등) 이식성을 높이는 훌륭한 선택이었습니다.
* **환경 인지적 설계**: 크로스 플랫폼(특히 WSL2 - Windows 연동)을 지원하는 K8s 툴을 만들 때는 `kubeconfig`의 경로 해석(`\\wsl$\...` 등)과 시스템 라이브러리(GUI 웹뷰)의 의존성 처리가 가장 까다로운 허들임을 인지해야 합니다.

## 2. 잘된 점 (What Went Well)
* **모듈화**: Cargo Workspace를 활용하여 의존성을 깔끔하게 분리했습니다.
* **UI/UX 디자인**: 단순한 기능 나열을 넘어 `frontend-design` 원칙을 적용하여 "Industrial Glassmorphism" 컨셉의 전문가용 모니터링 툴에 걸맞은 세련된 디자인을 구현했습니다.
* **비동기 통신 구조**: `kube-rs`의 `list_pods` 등 조회 기능과, 향후 Exec/Log를 위한 `tokio` 기반 비동기 스트리밍 뼈대를 선제적으로 잘 구성했습니다.

## 3. 잘못된 점 / 실수했던 부분들 (Mistakes & Pitfalls)
* **가장 치명적인 실수: 로컬 환경에 대한 무분별한 추측**
  * 사용자의 로컬 환경(시스템 라이브러리 설치 여부, K8s 클러스터 구동 여부)을 사전에 점검하거나 묻지 않고, 코드가 작성되었다는 이유만으로 "완료되었다"고 성급하게 보고했습니다.
* **도구 사용의 오용 (`cat <<EOF`)**
  * 긴 코드나 특수문자가 포함된 파일을 생성할 때 쉘 커맨드(`cat`)를 남용하다가 구문 오류(Syntax Error)가 발생했고, 이로 인해 `package.json`, `index.html` 등 필수 파일이 누락되는 대형 사고가 있었습니다.
* **버전별 API 변경 사항 추측**
  * `kube-rs 0.90` 버전의 API(`AttachParams` 등)를 사용할 때, 공식 문서나 소스코드를 철저히 확인하지 않고 기존 지식이나 이름으로 추측하여 여러 차례 빌드 에러를 유발했습니다.
* **Mock 데이터의 불명확성**
  * 실제 연결에 실패했을 때 보여주는 더미 데이터를 실제 파드 이름처럼 그럴싸하게 지어서, 사용자가 "연결이 성공한 것인지 실패한 것인지" 헷갈리게 만들었습니다.

## 4. 개선할 점 / 피드백 (Areas for Improvement)
* **Ask First, Act Later**: 환경 설정이나 의존성에 대해 불확실한 점이 있다면 임의로 진행하지 않고 사용자에게 질문하여 명확한 가이드를 받아야 합니다.
* **TDD / E2E 검증 필수**: 기능 구현 후에는 이번에 `kind` 클러스터를 띄우고 `test_kube.rs`로 검증했던 것처럼, **실제로 코드가 동작하고 통신하는지 스스로 검증한 뒤에 보고**해야 합니다.
* **신뢰성 있는 도구 사용**: 파일 조작은 에러 확률이 높은 Bash 스크립트 대신, 명확한 파일 쓰기 도구(`write_file`)를 적극 활용해야 합니다.

## 5. 실수하기 쉬운 부분에 대한 보완 제안 (Preventative Measures)
1. **Tauri GUI 의존성 문제**
   * **문제**: Tauri 앱은 리눅스/WSL 환경에서 컴파일 시 `webkit2gtk`, `libsoup` 등 수많은 시스템 데스크탑 라이브러리를 요구합니다. 헤드리스 환경에서는 100% 빌드가 실패합니다.
   * **보완**: `package.json` 스크립트나 `README.md`에 대상 OS별 필수 패키지 설치 명령어(`sudo apt install...`)를 반드시 명시하고, 빌드 전 스크립트로 체크하는 로직을 넣는 것이 좋습니다.
2. **ES Module과 Tailwind/PostCSS 설정 충돌**
   * **문제**: Vite 프로젝트(`"type": "module"`) 내에서 Tailwind 설정 파일을 `.js`로 만들고 `module.exports`나 `require`를 사용하면 컴파일이 조용히 무시되거나 에러가 납니다.
   * **보완**: Node.js 생태계에서 ESM과 CJS가 혼재될 때는 설정 파일의 확장자를 `.cjs`로 명시하거나, 전체를 `export default` (ESM) 형식으로 통일하는 습관을 들여야 합니다.
3. **비동기 스트림(`futures` vs `tokio`) 호환성**
   * **문제**: `kube-rs`는 `futures` 기반의 비동기 트레이트를 자주 반환하는데, 이를 `tokio` 런타임에서 처리하려다 타입 불일치(`AsyncRead` vs `AsyncBufRead`)가 잦습니다.
   * **보완**: `futures::io::AsyncBufReadExt`나 `tokio_util::compat` 같은 호환성 크레이트 및 확장 트레이트의 사용법을 명확히 숙지하고 있어야 합니다.
